# Retired Stage 2 attempts

This directory documents the two approaches we tried before settling on **Apache Jena Fuseki** as the SPARQL backend for Stage 2. The files are kept (not deleted) so future readers can see what was tried, what failed, and why — preventing the same dead ends being walked again. Strategic rationale for Stage 2 lives in `../../Ontology-Approach-to-Knowledge.md`. The post-mortem for both attempts is in the project plan.

## Attempt 1 — neosemantics (n10s)

**Why it looked attractive.** Single Neo4j plugin; no extra container; promised "SPARQL endpoint" in some blog posts and partial documentation.

**Why it didn't work.** n10s does not actually expose a SPARQL query endpoint. Its `/rdf/{db}/` paths provide:

- `/onto` — serialise an imported ontology to RDF
- `/cypher` — translate a Cypher query's result *to* RDF
- `/describe/*` — single-node RDF dumps

None of these accept a SPARQL string and return SPARQL JSON Results. Probing the endpoint (`/sparql`, `/query`, `/sparql/query`) returned 404 in every case. The `run_sparql` MCP tool would have 404'd every call.

**What we kept from this attempt.**

- `../uaf-mvo.ttl` — the OWL T-Box. Re-used by Fuseki unchanged.
- `../codegen/generate_mvo.py` — generates `uaf-mvo.ttl` from `:Stereotype` metamodel nodes. Still authoritative.
- The n10s constraint and `n10s.graphconfig.init` step in Neo4j (no longer in the runtime path; harmless).

## Attempt 2 — Ontop SPARQL endpoint + Neo4j JDBC driver

**Why it looked attractive.** Ontop is the canonical open-source OBDA stack — what the Ontology-Approach doc §5 actually describes. Neo4j JDBC v6 ships with a SQL→Cypher translator. On paper: SPARQL → Ontop SQL → Cypher.

**Why it didn't work.** A hard Java-version incompatibility between the only available pieces:

| Component | Java version |
|---|---|
| Ontop 5.x (including latest `5.5.0`) | Java 11 |
| Neo4j JDBC v6.x (the version with the SQL→Cypher translator) | Java 17 |
| Neo4j JDBC v5.x (Java 11 compatible) | No SQL→Cypher translator |

Concrete error on startup:

```
UnsupportedClassVersionError: org/neo4j/jdbc/Neo4jDriver has been compiled by
a more recent version of the Java Runtime (class file version 61.0), this
version of the Java Runtime only recognizes class file versions up to 55.0
```

(`61.0` is Java 17, `55.0` is Java 11.)

Bridging this would mean either building a custom Ontop image on a Java 17 base from scratch, or accepting that the older driver can't fulfil Ontop's SQL queries. Neither is a "Stage 2" sized commitment, and the resulting stack would still have known coverage limits for multi-hop SPARQL property paths.

**What we kept from this attempt.**

- Path A's architecture diagram informed Path B's choice: keep Neo4j as system of record, add a SPARQL sidecar.
- The `run_sparql` MCP tool body — the only change moving from Ontop to Fuseki was the URL default and the auth tuple.
